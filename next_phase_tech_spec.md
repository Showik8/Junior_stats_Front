# Junior Stats: მეორე ფაზის ტექნიკური და ბიზნეს სპეციფიკაცია

ეს დოკუმენტი დეტალურად აღწერს პროდუქტის სამომავლო განვითარების მოდულებს, მათი **ბიზნეს ლოგიკას** (პროდუქტის მენეჯერის გადმოსახედიდან) და **ტექნიკურ ლოგიკას** (მონაცემთა ბაზის არქიტექტურა, Backend სერვისები და Frontend კომპონენტები).

---

## 1. 🔍 სკაუტინგი და ტალანტების აღმოჩენა

### 1.1 მოთამაშის ციფრული პასპორტი (Digital ID)
**ბიზნეს ლოგიკა:**
მოთამაშის სრული ისტორიის ერთ სივრცეში თავმოყრა: ადრეული აკადემიები, ყველა ტრანსფერის ისტორია, სტატისტიკა სეზონების მიხედვით. სკაუტებს აქვთ წვდომა დეტალურ პროფილზე, რომელიც არის ვალიდური და გამორიცხავს ასაკის გაყალბების ან არაოფიციალურ ტიტულებზე მანიპულაციას.

**ტექნიკური ლოგიკა:**
- **DB Schema:** `Player` მოდელში რელაციების დამატება მოდელებთან: `TransferHistory` (from_club, to_club, date, status) და `SeasonStats`. უნიკალური გენერირებული `DigitalID`-ის (მაგ. ფედერაციის ინდექსი) დამატება `User`/`Player` ცხრილში.
- **Backend API:** 
  - `GET /api/players/:id/passport` - არჩევს მოთამაშის სრულ ისტორიას და სეზონურ სტატისტიკას ერთიანი JSON ობიექტის სახით.
- **Frontend:** 
  - ახალი `PlayerDigitalPassport` კომპონენტი. ტაიმლაინის (Vertical Timeline) UI, სადაც ქრონოლოგიურად არის ჩამოწერილი ტრანსფერები, ნათამაშები წუთები და მიღწევები სტატიკურ მონაცემებზე დაყრდნობით.

### 1.2 ვიდეო-ინტეგრაცია (Video Highlights)
**ბიზნეს ლოგიკა:**
მატჩის მნიშვნელოვანი მოვლენების (გოლები, იდეალური პასები, სეივები) ავტომატური დაკავშირება ვიდეო მასალის კონკრეტულ წამებთან. სკაუტი მოთამაშის პროფილში აჭერს "12 გოლი ამ სეზონში" და შეუძლია ზედიზედ ნახოს 12-ვე გოლის ვიდეო-ჭრა ერთი კლიკით.

**ტექნიკური ლოგიკა:**
- **DB Schema:** `MatchEvent` მოდელს დაემატება ველები: `videoUrl` (YouTube/Vimeo ან S3/R2 direct link) და `videoTimestampSeconds` (წამებში, მაგ. `3640` არის 60-ე წუთი და 40 წამი).
- **Backend API:** 
  - არსებული მატჩის მოვლენების API განახლდება ვიდეო ველების მხარდაჭერით.
  - `GET /api/players/:id/video-highlights` მოიძიებს ვიდეო ლინკის მქონე ყველა event-ს მოთამაშისთვის.
- **Frontend:** 
  - ვიდეო პლეერის (HTML5 ან YouTube Player iframe API) ინტეგრაცია მატჩის მოვლენების ტაიმლაინში. კლიკზე JavaScript პლეერს ავტომატურად რთავს მოვლენის `currentTime`-ზე.

### 1.3 გაფართოებული ანალიტიკა (Advanced Analytics & xG)
**ბიზნეს ლოგიკა:**
უფრო ღრმა მეტრიკების შემოღება ტრადიციული "გოლი/ასიტი"ს მიღმა (მაგალითად: xG - მოსალოდნელი გოლები, პასების სიზუსტე, სითბური რუკები). ეს სკაუტებს დაეხმარება სტატისტიკაზე დაფუძნებული, ობიექტური გადაწყვეტილების მიღებაში ეფექტურობის შეფასებით.

**ტექნიკური ლოგიკა:**
- **DB Schema:** `PlayerMatchStat` (მაკავშირებელი ცხრილი) მოდელს ემატება `xG` (Float), `passAccuracy` (Int), `rating` (Float), და `heatMapDataUrl` (String).
- **Backend API:** 
  - Background Job (Cron Scheduler), რომელიც მატჩის სტატუსის 'COMPLETED'-ზე გადასვლისას თვლის მოთამაშეთა რეიტინგებს ალგორითმით.
  - `GET /api/statistics/advanced` API დეშბორდისთვის.
- **Frontend:** 
  - Radar Charts და Bar Charts ინტეგრაცია (`Chart.js` ან `Recharts` ბიბლიოთეკებით), რათა მოთამაშეების პროფილები ვიზუალურად მდიდარი გახდეს.

## 2. 💰 კომერციალიზაცია და სპონსორების მოზიდვა

### 2.1 ციფრული სასპონსორო პაკეტები
**ბიზნეს ლოგიკა:**
ადმინისტრატორს შეუძლია ბანერების და სპონსორების ლოგოების მიბმა კონკრეტულ ასაკობრივ ლიგაზე ან სატურნირო ცხრილზე (მაგ: ამ ტურნირის სატურნირო ცხრილი წარმოდგენილია X ბრენდის მიერ). მონეტიზაციის მარტივი მოდელი.

**ტექნიკური ლოგიკა:**
- **DB Schema:** `Sponsor` მოდელის გაფართოება. რელაცია `PlacementInfo_T` ან Enum ველი: ბანერის პოზიცია (მაგ: `HOMEPAGE_HERO`, `TOURNAMENT_STANDINGS_TOP`, `MATCH_DETAILS_BANNER`).
- **Backend API:** 
  - თითოეული public route იძახებს `GET /api/sponsors?placement=X`-ს რომ მიიღოს აქტიური ბანერები. 
- **Frontend:** 
  - `SponsorBanner` Server/Client Component, რომელიც კონტექსტურად არენდერებს სპონსორის ვიზუალს სხვადასხვა პეიჯზე.

### 2.2 მატჩის მოთამაშე (Man of the Match - MOTM)
**ბიზნეს ლოგიკა:**
გულშემატკივრებს შეუძლიათ ხმის მიცემა მატჩის საუკეთესო მოთამაშეზე ლაივ რეჟიმში. MOTM ჯილდო პირდაპირ არის დაკავშირებული გენერალურ სპონსორთან.

**ტექნიკური ლოგიკა:**
- **DB Schema:** 
  - `MotmVote` ცხრილი (userId, matchId, playerId) - უნიკალური ინდექსით `userId` და `matchId` ზე, რომ 1 კაცმა 1 ხმა მისცეს.
  - `Match` მოდელში ემატება საბოლოო გამარჯვებული `motmPlayerId`.
- **Backend API:** 
  - `POST /api/matches/:id/vote-motm` (rate limiting-ით და JWT/Session ავთენტიფიკაციით).
- **Frontend:** 
  - ინტერაქტიული Voting შკალა (Poll UI) მატჩის "ლაივ" გვერდზე ანიმაციებით და შედეგების პროგრეს-ბარებით.

### 2.3 ბილეთები და მერჩანდაიზინგი (E-commerce ინტეგრაცია)
**ბიზნეს ლოგიკა:**
დამატებითი შემოსავალი Affiliate მარკეტინგით. გარე ტიკეტინგის პლატფორმებთან (TKT, Biletebi) და გუნდის პირად მაღაზიებთან ონლაინ დაკავშირება.

**ტექნიკური ლოგიკა:**
- **DB Schema:** `Match` მოდელში ემატება `ticketingUrl`. `Club` მოდელში ემატება `shopUrl`.
- **Backend API:** ამ ველების სერვალიზაცია Public API პასუხებში.
- **Frontend:** 
  - "ბილეთის ყიდვა" ან "ფანმერჩის ყიდვა" Call-to-Action ღილაკები მატჩის აფიშასა და გუნდის გვერდზე Affiliate (UTM) პარამეტრებით.

## 3. 🏟 გულშემატკივართა ჩართულობა (Fan Engagement)

### 3.1 გულშემატკივართა პროფილები (Fan Accounts) და Predictor
**ბიზნეს ლოგიკა:**
რიგით იუზერს რეგისტრაცია შეუძლია. იქმნება ფანების ქომიუნითი, რომლებსაც შეუძლიათ ტურის მატჩების ანგარიშების პროგნოზირება (Predictor) და რეიტინგში (Leaderboard) მონაწილეობა. ზრდის აპლიკაციაში დაბრუნების (Retention) მაჩვენებელს.

**ტექნიკური ლოგიკა:**
- **DB Schema:** 
  - `User` მოდელში `role` Enum-ს ემატება `FAN`.
  - ახალი `Prediction` მოდელი (userId, matchId, predictedHomeScore, predictedAwayScore, pointsAwarded).
- **Backend API:** 
  - მატჩის შედეგის შეყვანისას Trigger/Service ადარებს რეალურ შედეგებს პროგნოზებს და არიგებს 1, 3 ან 5 ქულას ლოგიკის მიხედვით.
  - `GET /api/fans/leaderboard`.
- **Frontend:** 
  - Fan Dashboard-ის UI მარშრუტი (app/fan) და პროგნოზების შესაყვანი ინტერფეისები.

### 3.2 მეტოქეების შედარება (Head-to-Head - H2H)
**ბიზნეს ლოგიკა:**
ინტერაქტიული მოდული მაყურებლებისთვის მოსალოდნელი მატჩის წინ. ორი გუნდის ისტორიული ურთიერთშეხვედრების სტატისტიკის, ბოლო 5 მატჩის ფორმის მარტივი შედარება.

**ტექნიკური ლოგიკა:**
- **Backend API:** 
  - `GET /api/teams/compare?team1=:id&team2=:id`. 
  - Advanced Prisma Query: ეძებს `Match` მოდელში, სადაც team1 და team2 იყო homeTeam ან awayTeam, და ითვლის Win/Draw/Loss სტატისტიკას.
- **Frontend:** 
  - გამოყოფილი H2H გვერდი დინამიური პროგრეს-ბარებით, რომელიც ვიზუალურად აჩვენებს ვის აქვს უპირატესობა.

### 3.3 Push-შეტყობინებები ლაივ-მოვლენებზე
**ბიზნეს ლოგიკა:**
მომხმარებელი "იწერს" (subscribe) გუნდს და ტელეფონზე იღებს Notification-ს პირდაპირ ეთერში, როცა ტარდება მატჩი და გადის გოლი ან ფიქსირდება წითელი ბარათი.

**ტექნიკური ლოგიკა:**
- **Backend API:** 
  - Push სერვისის გაშვება (Pusher, Firebase Cloud Messaging, ან Server-Sent Events). 
  - როცა Admin იძახებს მატჩის ივენთის შექმნის API-ს (მაგ. Goal), Backend ავტომატურად ისვრის event payload-ს Pub/Sub არხში.
- **Frontend:** 
  - PWA Service Worker კონფიგურაცია, Request Notification Permission ლოგიკა და ეკრანზე Popup-ბის ჩვენება შესული შეტყობინებისას.

## 4. ⚖️ ფედერაციის ოპერაციული მართვა და გამჭვირვალობა

### 4.1 მსაჯების პორტალი (Referee Dashboard)
**ბიზნეს ლოგიკა:**
მსაჯებისთვის განკუთვნილი PWA ან მობილურზე მორგებული პანელი. მატჩის შემდეგ მსაჯი პირდაპირ ტელეფონზე ავსებს ოქმს (და არა ფურცელზე) და ფურცლოვანი ბიუროკრატიის გარეშე, მონაცემები ეგრევე ხდება საჯარო.

**ტექნიკური ლოგიკა:**
- **DB Schema:** 
  - მოძრავ `User` ექნება `role = REFEREE`. `Match` მოდელს ემატება `mainRefereeId`, `assistantRefereeIds`.
- **Frontend:** 
  - მობილურ ეკრანებზე სრულად მორგებული UI მსხვილი ღილაკებით (Touch-friendly form). 
  - Offline დასტური (Service Worker / IndexedDB-ში შენახული ოფლაინ request-ები), რათა ცუდ ინტერნეტიან სტადიონზე ოქმი არ დაიკარგოს და ინტერნეტში დაბრუნებისას Sync მოხდეს Backend-თან.

### 4.2 დისციპლინური მოდული და სასჯელების ავტომატიზაცია
**ბიზნეს ლოგიკა:**
ჭკვიანი ლოგიკა, რომელიც ადამიანურ შეცდომებს გამორიცხავს. თუ მოთამაშემ მიიღო 3 ყვითელი ან 1 წითელი ბარათი წინა თამაშებში, სისტემა ავტომატურად ბლოკავს კლუბის ადმინს, რომ ეს მოთამაშე შემდეგი მატჩის განაცხადში (Roster) ვერ შეიყვანოს. 

**ტექნიკური ლოგიკა:**
- **DB Schema:** 
  - `DisciplinaryAction` ან `Suspension` ცხრილი (playerId, reason, suspendedMatchIds).
- **Backend API:** 
  - Rule Engine Middleware. `Match Roster`-ის შენახვისას API გაივლის შემოწმებას `checkPlayerEligibility()`, რომელიც ეძებს წინა გაფრთხილებებს და თუ იპოვა აბრუნებს HTTP 403 / 400.
- **Frontend:** 
  - Tournament Club Admin-ის გუნდის შედგენის ფორმაში ეს კონკრეტული Player და-disable-ებულია ვიზუალურად + აჩვენებს Tooltip-ს: "დისკვალიფიკაცია (წითელი ბარათი)".

### 4.3 სამედიცინო და ტრავმების რეესტრი
**ბიზნეს ლოგიკა:**
მონაცემებზე დაფუძნებული სამედიცინო პოლიტიკა. კლუბის ექიმები აფიქსირებენ ტრავმებს, ხოლო ფედერაციას აქვს ანალიტიკა: რომელი ასაკი, რომელ თვეში და რა ტიპის მოედანზე (ხელოვნური/ბუნებრივი) უფრო მიდრეკილია ტრავმებისკენ. ეს ინფორმაცია დაცულია და არააქტიურია პუბლიკისთვის მოთამაშის უფლებების დასაცავად.

**ტექნიკური ლოგიკა:**
- **DB Schema:** 
  - ახალი `InjuryReport` მოდელი (playerId, matchId?, injuryType, severity, estimatedRecoveryDays, fieldType, reportDate).
- **Backend API:** 
  - Data Privacy (მხოლოდ დაშვებულ Admin/Doctor Role-ებს შეუძლიათ კონკრეტული PlayerId-ის ამოღება).
  - Aggregate წამოღება ადმინ დეშბორდისთვის (მაგ. ტრავმების % ხელოვნურ საფარზე vs ბუნებრივზე).
- **Frontend:** 
  - დაფარული Doctor/SuperAdmin მოდული ახალი form-ებით და სამედიცინო Metrics დეშბორდით.
